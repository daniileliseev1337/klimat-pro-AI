## CRITICAL

### C1. «Моя доля» на карточке проекта показывает долю ВЛАДЕЛЬЦА любому зрителю (= замечание A владельца)
- **Где:** `src/App.jsx:2776–2790` (индикатор на карточке в `Projects`).
- **Что:** расчёт всегда `mine = contract − sum(доли участников)` — это **остаток владельца**.
  Участник-юзер видит карточку доступного проекта и подпись «Моя доля: 21 000 ₽ (60%)»,
  хотя ЕГО доля 14 000 ₽ (40%). RLS (`project_shares_select`) отдаёт участнику ровно его
  строку доли — данные для правильного отображения УЖЕ приходят на клиент в `sharesByProject`.
- **Как чинить:** ветвление по зрителю: `p.ownerId === profile.id` → текущий расчёт остатка;
  иначе → найти в `shs` строку `participantUserId === profile.id` и показать
  `shareToAmount(share, contract)` + процент (`share_kind==='percent' ? share_value :
  amount/contract*100`); нет своей строки → индикатор не показывать.
  То же ветвление продумать для ProjectForm-секции «Доли участников» (форму открывает
  владелец/админ, так что менее срочно).

### C2. Уведомления о комментариях ПРОЕКТА не работают вовсе
- **Где:** фронт `src/App.jsx:4759` (`CommentsSection.send` → `sendPush(client,"comment",null,{projectId,...})`),
  edge `web-push-notify/index.ts:204–216` (ветка `comment`).
- **Что:** edge-ветка `comment` требует `taskId` (uuid) и возвращает 400; фронт шлёт `projectId`
  без `taskId`. Ошибка проглатывается `sendPush` (best-effort) → **ни push, ни inbox-строки
  никому** при любом комментарии к проекту. Ветка написана под комментарии ЗАДАЧ, но
  task-комментарии шлют `task_question` (см. C3) — т.е. ветка не обслуживает никого.
- **Как чинить:** в edge ветке `comment` поддержать `projectId`: адресаты = владелец проекта +
  `project_members` + executors-с-userId (через `projects.executors`), минус инициатор;
  body с именем проекта; url `/projects/<id>`; флаг `notif_comment`.

### C3. Workflow ТЗ задач (6.4b) не шлёт уведомлений — типы не реализованы в edge
- **Где:** фронт шлёт `task_tz_proposed` (App.jsx:3538), `task_tz_approved`/`task_tz_rejected`
  (3549–3550), `task_question` (3568). Edge обрабатывает только task_assigned/task_status/
  task_created/deadline/project_taken/team_invite/comment/project_published → эти четыре
  падают в `unknown type` → **ни push, ни inbox**.
- **Что ломает:** вторая сторона не узнаёт о предложении изменения ТЗ / решении / вопросе,
  пока сама не откроет задачу. Блокирующий апрув ТЗ без сигнала = подвисшие задачи.
- **Как чинить:** добавить 2 ветки в edge: tz-события (база: author+assignee минус инициатор,
  body «Предложено изменение ТЗ: <title>» / «ТЗ принято/отклонено», url /tasks, флаг notif_task)
  и `task_question` (та же база, флаг notif_comment).

### C4. Админ-редактирование чужого проекта молча ПЕРЕЗАПИСЫВАЕТ владельца на админа
- **Где:** `src/App.jsx:140–164` (`projectJsToDb` всегда пишет `owner_id: ownerId`),
  `updateProject` (380), вызов `saveProject` в Projects с `ownerId = profile.id` (2596, 7455).
  Кнопка ✏️ доступна админу на чужих проектах (2938).
- **Что:** админ открывает чужой проект (поправить опечатку) → `update {..., owner_id: admin.id}`
  → проект «угнан»: настоящий владелец теряет его из своих фильтров/дашборда/долей/платежей
  (RLS на shares/payments привязан к owner). Причём после смены владельца `set_project_shares`/
  `set_project_payments` для админа уже проходят — угон работает end-to-end без ошибок.
  (Если живая RLS-политика projects запрещает менять owner_id — тогда админ просто не может
  сохранить чужой проект, что тоже баг, но менее страшный. RLS projects живёт вне репо-миграций —
  проверить на живой БД.)
- **Как чинить:** в `updateProject` НЕ слать `owner_id` (выделить из projectJsToDb; insert —
  оставить). Плюс решить поведение админа для долей/платежей чужого проекта: либо скрывать
  секции, либо ослабить RPC-гейты до `owner OR is_admin()`.

### C5. Импорт бэкапа ТЕРЯЕТ оплаты (и бэкап неполный)
- **Где:** `BackupPanel` (App.jsx:6561–6676): export = `{projects, txs}`;
  import → `insertProjectsBulk` → `projectJsToDb` (без `paid_amount` — намеренно убран
  в истории платежей, источник правды — `project_payments`).
- **Что:** «страховка» обещает восстановление, но: `project_payments` не экспортируются;
  при импорте `paid_amount` вставится 0 и триггер держит его 0 (платежей нет) → все
  «Оплачено» обнуляются. Также не бэкапятся shares, tasks, clients.
- **Как чинить:** включить в экспорт payments+shares (+опц. clients/tasks); при импорте
  восстанавливать платежи через `set_project_payments` (доли — `set_project_shares`).
  Минимум — честно написать в UI, что бэкап покрывает только проекты и кошелёк БЕЗ оплат.

## MAJOR

### M1. Sticky-шапка перекрывает панель уведомлений (= замечание E владельца)
- **Где:** sticky-контейнер `App.jsx:7205` (`zIndex:50`); шапка `7216` и полоса вкладок `7393` —
  обе с `backdropFilter: blur(8px)`; панель `NotificationBell.jsx:92–104` (`zIndex:80`,
  desktop `absolute`, mobile `fixed` с `top` замеренным один раз).
- **Почему:** `backdrop-filter` создаёт **stacking context + containing block** для
  fixed/absolute потомков. Панель живёт внутри stacking context шапки → её zIndex 80 заперт;
  полоса вкладок — более поздний sibling со своим контекстом → рисуется ПОВЕРХ панели
  (тёмная полоса режет список — то, что на скрине). На мобильном хуже: `position:fixed`
  позиционируется не от вьюпорта, а от шапки, и `top` не пересчитывается при скролле.
- **Как чинить:** рендерить панель через `createPortal(..., document.body)` с `position:fixed`
  (координаты от `getBoundingClientRect()` колокольчика; на мобильном left:8/right:8 как сейчас),
  `zIndex` ~90 (ниже Modal=100, выше sticky=50). Закрывать панель на scroll/resize или
  пересчитывать top. Полумера «zIndex шапке выше вкладок» мобильный containing block не чинит —
  портал правильнее.

### M2. takeProject не синхронизирует executors (= база замечания C владельца)
- **Где:** `App.jsx:2997–3003` («Взять в работу»): пишет `executor` (строку) прямым update,
  но НЕ добавляет `{name, userId}` в `executors` jsonb и НЕ добавляет в `project_members`.
- **Что:** при следующем сохранении формы владельцем `projectJsToDb` пересоберёт `executor`
  из `executors` → имя взявшего ПОТЕРЯЕТСЯ. И исполнитель не в команде (замечание C:
  «исполнитель автоматически в команде с ролью editor»).
- **Как чинить (вместе с замечанием C):** при take — append в `executors` jsonb + insert
  в `project_members (role=editor)`; при назначении исполнителей в ProjectForm — синхронизация
  executors↔members (добавление; удаление — решить с владельцем). Лучше в RPC `take_project`
  на сервере, чем 3 клиентских запроса.

### M3. ReportViewer: старый бренд + дублированная разметка + XSS через innerHTML
- **Где:** `App.jsx:6746–6781` (print-root через `innerHTML`) и `6816–6891` (то же JSX-ом).
- **Что:** (1) шапка отчёта — **«ДАНИИЛ — Отчёт по проектам»** (6750, 6822) — старый бренд,
  сайт давно КЛИМАТ-ПРО (это и есть «шапка в стиле сайта» из замечания #4); (2) разметка
  отчёта продублирована дважды (innerHTML для печати + JSX для предпросмотра) — при редизайне
  #4 сводить к одному источнику; (3) `innerHTML` подставляет `${p.name}`, `${p.notes}`,
  `${p.client}` БЕЗ экранирования → stored XSS: любой approved-юзер может назвать проект
  `<img src=x onerror=...>`, владелец/админ откроет отчёт → исполнение. Закроется само,
  если редизайн #4 уйдёт от innerHTML на чистый JSX (React экранирует) — учесть как
  требование редизайна.

### M4. Realtime-канал задач без серверного фильтра + клиентский фолбэк
- **Где:** `App.jsx:3938–3966` — подписка на ВСЕ изменения `project_tasks` (без `filter`),
  приватность держится на серверной RLS (WALRUS) + клиентском `canSeeRow`.
- **Что:** работает, но `canSeeRow` для проектных задач пускает строку, если проект в списке

  Косметика+перф; не дыра. При редизайне TasksView (#2) — учесть.
