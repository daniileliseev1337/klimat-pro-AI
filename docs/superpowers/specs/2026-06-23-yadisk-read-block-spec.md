## Три трактовки «Я.Диск считывание» (выбрать одну)

### Вариант A — READ-ONLY зеркало рабочей папки (рекомендую)
Дашборд показывает содержимое рабочей папки проекта на Я.Диске (список файлов/папок, размеры, даты,
ссылки на скачивание/просмотр). Привязка «папка Я.Диска ↔ проект дашборда». Не копирует — читает.
- **Польза:** видишь реальные файлы проекта (чертежи, ПЗ, расчёты) прямо в карточке проекта.
- **Изолированность:** новый компонент `YandexDiskFiles` + новая Edge Function `yadisk-read` +
  поле `projects.yadisk_path`. НЕ трогает существующий `ProjectFiles`/Nextcloud.

### Вариант B — Watcher изменений (#3 из очереди)
Авто-отслеживание новых/изменённых файлов в рабочей папке → лента событий / уведомление в дашборде
(«в проект X добавлен чертёж Y»). Сложнее (polling/ревизии Я.Диск API, дедуп, хранение состояния).

### Вариант C — Импорт Я.Диск → Nextcloud
Копировать выбранные файлы с Я.Диска в Nextcloud-хранилище проекта (разовый импорт/бэкап).

## Архитектура (для A, рекомендуемого)

Два способа дотянуться до Я.Диска `danii1126`:
1. **Я.Диск REST API** (`cloud-api.yandex.net/v1/disk/resources`) — OAuth-токен приложения.
   Edge Function `yadisk-read`: `GET /resources?path=<папка>` → список файлов (name, size, modified,
   mime, preview, file-download-url). Токен — в `config.json` на сервере (как креды Nextcloud), НЕ в git.
   - Плюс: работает с прода (VPS→API Яндекса), не зависит от того, включён ли ПК.
   - Минус: видит облачную копию; нужен OAuth-токен (получить разово на oauth.yandex.ru).
2. **Локальный агент на ПК** читает `F:\Работа\<проект>` напрямую (FS) и пушит индекс в Supabase.
   - Плюс: без токена Я.Диска, видит локальные файлы сразу.
   - Минус: работает только когда ПК включён; нужен фоновый процесс (как Telegram-бот).

Рекомендация: **REST API** (способ 1) — надёжнее и не зависит от ПК.

## План реализации варианта A (когда будут ответы)

1. Миграция БД: `ALTER TABLE projects ADD COLUMN yadisk_path text;` (путь папки на Я.Диске).
2. Edge Function `yadisk-read` (по образцу `nextcloud`): action `list` (path) → REST API Я.Диска
   с токеном из `config.json`; action `download-url` (path) → временная ссылка. RLS: только участники проекта.
3. Фронт: компонент `YandexDiskFiles({projectId, path})` — таблица файлов (иконка/имя/размер/дата/скачать),
   как `ProjectFiles`, но read-only. Вкладка/секция в карточке проекта. Поле ввода `yadisk_path` (для владельца).
4. Деплой edge + БД-миграция (транзакционный тест) + web.
5. Верификация: владелец привязывает путь к проекту → видит реальные файлы.

## Почему так (честно)

Реализовать «ещё один блок» наобум ночью значило бы: выдумать требования (Karpathy: не выдумывать;
трактовки показать, не выбирать молча), рискнуть только что задеплоенным редизайном, и почти наверняка
сделать не то (Я.Диск vs Nextcloud, нет токена). Грамотнее — этот спек: завтра 3 ответа → точная реализация.
