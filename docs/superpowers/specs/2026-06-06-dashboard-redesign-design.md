# Переработка дашборда (этап 6.5) — дизайн

**Дата:** 2026-06-06
**Проект:** daniil-dashboard v3.0 (React + Vite + self-hosted Supabase)
**Статус:** утверждён, готов к writing-plans
**ТЗ:** разделы 4.3 / 6.5 («живой сводный экран»); разд. 8 («набор метрик согласовать с владельцем» — выполнено в этом brainstorming).

## Проблема

ТЗ описывает дашборд как «статичные плашки», но фактически компонент `Dashboard`
(`src/App.jsx` ~2047) **уже живой**: 4 KPI (включая баланс месяца с трендом),
пирог по стадиям, бар-чарт доходы/расходы за 6 мес, списки просроченных и ближайших
дедлайнов, блок «Финансы месяца» с прогресс-баром — всё на `recharts`.

Поэтому 6.5 — это **доработка**, а не создание с нуля. Владелец выбрал улучшение по
всем четырём векторам (больше метрик, интерактивность, перекомпоновка, задачи на
дашборде) с **сбалансированным** фокусом (без одного доминирующего hero-блока).

## Цели и не-цели

**Цели:**
- Добавить недостающие метрики: дебиторка, расходы по категориям, накопительный баланс.
- Сделать финансовую часть управляемой периодом (Месяц/Квартал/Год/Всё).
- Втянуть на сводный экран операционку — «мои задачи».
- Перекомпоновать всё в осмысленные зоны.
- Drill-down с дашборда в раздел «Проекты».

**Не-цели (вне scope, сознательно отвергнуто владельцем):**
- Выручка по типам проектов (мало типов/данных).
- Realtime live-обновление дашборда (данные свежи при заходе; выгода спорная).
- Лента активности (дублирует уже готовый Центр уведомлений).
- Изменения БД, edge-функций, других вкладок.

## Ключевые решения (из brainstorming)

1. **Дашборд персональный.** Показывает проекты и личные финансы владельца под
   действующей RLS. Не агрегат по команде КЛИМАТ-ПРО.
2. **Каркас «Зоны по смыслу».** Период-селектор → KPI-полоса → три именованные секции:
   «⚠ Требует внимания», «💰 Финансы», «📁 Проекты». (Отвергнуты bento-сетка и
   двухколоночная раскладка — хуже иерархия / мобильная адаптация.)
3. **Согласованный набор виджетов** (см. ниже). Текущие виджеты сохраняются и
   перекомпоновываются; старый блок «Финансы текущего месяца» (прогресс-бар) **удаляется**
   как дублирующий KPI-баланс + графики + период.
4. **Период управляет только финансовой зоной** (+ KPI-баланс). Дедлайны, задачи, пирог
   стадий, дебиторка — «на сейчас», от периода не зависят.
5. **«Мои задачи» = назначенные мне** (`assignedTo === profile.id`), статусы кроме
   «Готово»/«Отменена».
6. **Без миграций БД.** Все новые метрики считаются на клиенте из уже загруженных
   `projects` / `txs`; задачи берутся существующим RPC `get_tasks`.

## Раскладка (зоны)

```
┌───────────────────────────── Дашборд ──────────────[ Месяц | Кв | Год | Всё ]┐
│ [KPI Активных🔗] [KPI Портфель] [KPI Получено] [KPI Баланс за период ↑]       │
│                                                                              │
│ ⚠ ТРЕБУЕТ ВНИМАНИЯ                                                            │
│   [Просроченные дедлайны проектов]      [Мои задачи · просрочено/сегодня]     │
│                                                                              │
│ 💰 ФИНАНСЫ · за выбранный период                                             │
│   [Доходы/расходы (бар)] [Накопит. баланс (линия)] [Расходы по категориям]   │
│   [────────── Дебиторка · жду оплат: Σ; список по убыванию ──────────]       │
│                                                                              │
│ 📁 ПРОЕКТЫ                                                                    │
│   [Пирог по стадиям 🔗]                  [Ближайшие дедлайны]                 │
└──────────────────────────────────────────────────────────────────────────────┘
```

На мобильном (`useIsMobile`, ≤640px) секции и карточки стекаются в одну колонку —
существующие сетки `repeat(auto-fit, minmax(...))` это уже обеспечивают.

## Виджеты — спецификация

### KPI-полоса (4 карточки, `KpiCard`)
| Карточка | Значение | Источник | Период |
|---|---|---|---|
| Активных проектов | `active.length`, sub «всего: N»; **кликабельна** → drill-down | `projects` | нет |
| Портфель | `totalContract` (Σ contractSum по не-архивным) | `projects` | нет |
| Получено | `totalPaid`, sub «осталось: …» | `projects` | нет |
| Баланс за период | доходы − расходы за период; тренд vs предыдущий аналогичный период | `txs` | **да** |

### ⚠ Требует внимания
- **Просроченные дедлайны проектов** — как сейчас (`overdue`), но подпись «просрочен N дн».
- **Мои задачи** — НОВОЕ. Заголовок со счётчиками «просрочено N · сегодня M».
  Список: сперва просроченные (`dueDate < today`), затем сегодняшние (`dueDate === today`).
  Источник — `tasks` (мои активные). Пусто → «Нет горящих задач».

### 💰 Финансы (зависит от периода)
- **Доходы/расходы (бар)** — существующий `BarChart`, временной ряд по периоду
  (адаптивная гранулярность, см. ниже).
- **Накопительный баланс (линия)** — НОВОЕ. `LineChart`/`Line` (уже импортированы).
  Кумулятивная сумма (доходы−расходы) по точкам периода.
- **Расходы по категориям (donut)** — НОВОЕ. `PieChart` по `EXPENSE_CATS`, только расходы
  за период, сортировка по убыванию; мелкие категории сворачиваются в «Прочее» при переполнении легенды.
- **Дебиторка** — НОВОЕ, широкая карточка. Список не-архивных проектов с остатком
  `contractSum − paidAmount > 0` по убыванию (топ-N + «ещё K»), итог «жду оплат» = Σ остатков.
  Не зависит от периода.

### 📁 Проекты
- **Пирог по стадиям** — существующий, + **drill-down**: клик по сектору → раздел «Проекты»
  с фильтром этой стадии.
- **Ближайшие дедлайны** — как сейчас (`upcoming`, топ-4).

## Данные и data flow

- **Загрузка задач на верхний уровень.** В `App` добавляется состояние `tasks` (рядом с
  `projects`/`txs`, ~App.jsx:6713) и загрузка через существующую `fetchTasks(supabase,
  { assignedTo: profile.id })`. Проп `tasks` передаётся в `Dashboard` (вызов ~App.jsx:7183).
- **Props `Dashboard`** расширяются: `{ projects, txs, tasks, onDrillStage }`
  (`clients`/`profile` уже передаются). `profile` нужен для подписи/фильтра при желании.
- **Все новые расчёты — на клиенте** (дебиторка, расходы по категориям, cash flow,
  агрегаты периода) из `projects`/`txs`. RLS уже отдаёт только данные владельца. Без новых RPC.

## Период (сквозная логика)

- Состояние `period` в `Dashboard`: `month | quarter | year | all`. Дефолт — `month`
  (текущее поведение).
- Хелпер диапазона `[from, to)` от «сейчас». Транзакция входит в период по строковому
  `t.date` (формат `YYYY-MM-DD`).
- Влияет на: KPI-баланс, бар доходы/расходы, линию cash flow, donut расходов.
- **Адаптивная гранулярность** временных рядов: `month` → по дням, `quarter` → по неделям,
  `year`/`all` → по месяцам. **Fallback:** если в периоде слишком мало точек/данных,
  деградирует к помесячной разбивке (решается при реализации, не усложнять преждевременно).

## Drill-down навигация

- `Dashboard` получает колбэк `onDrillStage(stage)` (и для KPI «Активных» — фильтр
  «Активные»). Колбэк в `App` делает `setTab("projects")` + устанавливает начальный фильтр.
- `Projects` сейчас держит `stageFilter` во внутреннем state (App.jsx:2285). Для drill-down
  фильтр поднимается/прокидывается начальным пропом (механизм — на этап writing-plans;
  переиспользуем существующий паттерн навигации `onNavigate`/`setTab`).

## Вне scope / что не трогаем

Вкладки Проекты/Финансы/Задачи (кроме подъёма `tasks` и начального фильтра), БД,
edge-функции, уведомления, аутентификация. Realtime, выручка по типам, лента активности.

## Критерии приёмки (verification)

1. `npm run build` — зелёная сборка.
2. Период-пилюли переключают **только** финансовую зону и KPI-баланс; остальное стабильно.
3. Дебиторка = Σ остатков (`contractSum−paidAmount`) по не-архивным с остатком > 0; список отсортирован.
4. Расходы по категориям: суммы по `EXPENSE_CATS` за период совпадают с данными `txs`.
5. Накопительный баланс монотонно отражает кумулятивную сумму по точкам периода.
6. «Мои задачи» приходят на дашборд, фильтруются «мои/активные», корректно делятся на
   просрочено/сегодня.
7. Drill-down: клик по сектору пирога / KPI «Активных» открывает «Проекты» с нужным фильтром.
8. Мобильная раскладка (≤640px): секции стекаются, ничего не уезжает за край.
9. Старый блок «Финансы месяца» отсутствует; дублирования метрик нет.

## Заметки реализации / риски

- **Объём:** фронт-only, без миграций. Основная работа — внутри `Dashboard` + точечно
  подъём `tasks` и начального фильтра в `App`. `Dashboard` уже немал — при росте вынести
  под-виджеты (Дебиторка, РасходыПоКатегориям, CashFlow, МоиЗадачи) в отдельные компоненты.
- **Инлайн-стили + медиа-запросы не работают** (известная грабля проекта) — мобильную
  адаптацию держать на `auto-fit/minmax` и `useIsMobile`, не на CSS media в style.
- **Гранулярность** — главный источник переусложнения; начинать с простого помесячного
  ряда, адаптивность вводить только если визуально оправдано.
- Деплой по стандартной процедуре: `npm run build` → `deploy/nextcloud/deploy-web.sh`;
  iOS PWA — холодный перезапуск после деплоя.
