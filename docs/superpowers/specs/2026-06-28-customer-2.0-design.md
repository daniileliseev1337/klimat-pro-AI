## 1. Цель

Дать заказчику (аккаунт, привязанный к записи `clients`) активную роль в своих проектах:
видеть и ставить задачи, общаться по ним (комментарии/ТЗ), принимать работу, и самому
инициировать проекты. Сохранить безопасность: заказчик НЕ получает доступ к внутренним
данным (финансы, доли, notes, чужие проекты).

## 2. Решения грилинга (зафиксированы с владельцем)

| # | Ось | Решение |
|---|-----|---------|
| 1 | Видимость задач | Заказчик видит **все** задачи своих проектов |
| 2 | Комментарии/ТЗ | Видит и пишет (канал общения с заказчиком); приватной переписки нет — уйдёт в будущий мессенджер |
| 3 | Назначение исполнителя | Кнопка «Взять в работу» уже есть (смена статуса); автоподтяг исполнителя из `executors` проекта |
| 4 | Создание проектов | Оба режима: быстрая заявка / подробная; через **`project_requests`** (см. §5) |
| 5 | Права на статусы | **Приёмка**: заказчик На проверке→Готово / вернуть→В работе; производственные статусы — исполнитель |
| 6 | **Подбор исполнителя заказчиком (v2)** | В заявке заказчик выбирает путь: **«на поиск исполнителя» (маркетплейс)** ЛИБО **прямое назначение** конкретного исполнителя из списка. Проект из заявки материализует **сотрудник** (кнопка «Принять») |

## 3. Факты БД (проверено на self-hosted, 2026-06-28 / 2026-06-29)

- `projects.owner_id` — **NOT NULL** → проект-заявка обязан иметь владельца.
- `projects.client_id` — nullable; `projects.stage` — text **без CHECK** (свободные значения).
- `projects.visibility` — CHECK `private|team|selected|marketplace`.
- **Два слоя ролей** (важно, проверено 2026-06-29):
  - `profiles.role` — только `admin`/`user` (грубый слой).
  - **`user_roles(user_id, role)`** — тонкий слой, CHECK `employee|client|visitor`. Фронт оперирует
    ИМ (`App.jsx:8778` `myRoles.includes('employee'|'client')`). Данные: 6×`employee`, 1×`client`,
    причём этот `client` (`6d95d516`) имеет ОБЕ роли `{client,employee}` — **гибрид** (и заказчик,
    и сотрудник). PK `(user_id, role)`; RLS select = `user_id=uid OR is_admin()`.
  - «Заказчик» = есть роль `client` (≡ привязка `clients.user_id`, их синхронизирует
    `set_client_user`). «Сотрудник» = есть роль `employee`. Аккаунт может быть и тем, и другим.
- **`set_client_user(p_client_id, p_user_id)`** при привязке: добавляет роль `client` в `user_roles`
  И ставит `profiles.approved=true`, `access_requested=false` (миграция 0005) → **инвариант И-2
  уже обеспечен** существующим кодом.
- **Чистого заказчика (роль `client` БЕЗ `employee`) в БД сейчас НЕТ** — единственный client гибрид.
  Для тестов сценариев заказчика нужно создать тестовый аккаунт `client` без `employee` (Задача 0).
- `is_approved()` = `profiles.approved = true` **без проверки роли** → заказчик (`approved`) проходит его.
- `am_i_client()` = `exists(clients where user_id = auth.uid())`.
- `is_project_client(p)` = проект.client_id → `clients.user_id = auth.uid()` (миграция 0007).
- `am_i_client()`, `is_project_client(p)`, `get_my_client_projects()` (безопасная проекция без
  `owner_id`/`notes`/долей) — существуют (миграция 0007).
- **`is_employee()` НЕ существует** — вводится этой фазой (см. §4.0).
- **Маркетплейс готов:** проект `visibility='marketplace'` + `stage='Поиск исполнителя'` рассылает
  broadcast-уведомление (флаг `profiles.notif_new_project`, default true); исполнитель сам берёт
  через RPC `take_project` (попадает в `executors` jsonb + `project_members(editor)`).
- `clients.user_id` — привязка аккаунта заказчика к записи. **Данные: 1 аккаунт-заказчик → 1 запись
  `clients`** (проверено 2026-06-29). Множественности нет → §8.1 закрыт.
- `profiles` для списка исполнителей: `name`, `position` (должность), `username`. Контактов
  (email/telegram_chat_id) и `role` заказчику НЕ показывать.

## 4. Новая функция доступа + архитектура задач

### 4.0. `is_employee()` (вводится фазой) — на роли `employee`, НЕ на `NOT am_i_client`
```sql
create or replace function public.is_employee() returns boolean
language sql stable security definer set search_path = public, pg_temp as $$
  select public.is_approved()
     and exists (select 1 from public.user_roles
                 where user_id = auth.uid() and role = 'employee');
$$;
grant execute on function public.is_employee() to authenticated;
```
**Почему по роли, а не `NOT am_i_client()` (ревью + данные 2026-06-29):** аккаунт может быть
гибридом `{client,employee}` (реальный кейс `6d95d516`). `NOT am_i_client()` отрезал бы такого
СОТРУДНИКА от прав. Роль `employee` в `user_roles` — точный признак сотрудника, согласован с
фронтом. «Сотрудник» = одобренный аккаунт с ролью `employee`. Чистый заказчик (`client` без
`employee`) → `is_employee()=false`. Применяется к RLS заявок (§5.2), `list_available_executors()`
(§5.4), marketplace-веткам (§7).

### 4.1. Базовый доступ к задачам — одна правка
В `can_access_project_comments(p_project_id)` добавить ветку `or public.is_project_client(p_project_id)`.
Функция централизованно гейтит RLS задач (SELECT/INSERT/UPDATE-check), комментариев и ТЗ →
заказчик автоматически получает: чтение задач, **создание** задач (INSERT-политика уже
требует `is_approved AND author=uid AND can_access_project_comments`; заказчик `approved`),
чтение/запись комментариев и ТЗ своих проектов.

**Обоснование централизации:** баг 6.2 (отчёт 0626) возник из-за дублирования логики в двух
функциях, которые разъехались. Расширяем ОДНУ функцию — меньше рассинхрон-риск.

**⚠ ТЗ — отдельный гейт (ревью B-1, 2026-06-29).** Запись ТЗ идёт ТОЛЬКО через
`propose_tz_version()` (SECURITY DEFINER; прямой INSERT в `task_tz_versions` заблокирован RLS —
есть только SELECT-политика). Её внутренний гейт: `author_id=uid OR assigned_to=uid OR is_admin
OR is_project_editor`. Заказчик к задаче, поставленной **сотрудником**, не подходит ни под одно
условие → `forbidden`. Поэтому `can_access_project_comments` НЕ достаточно: **в
`propose_tz_version()` добавить `OR public.is_project_client(t.project_id)`**, иначе «заказчик
пишет ТЗ» сломано для задач сотрудника (большинство).

**⚠ Валидация `assigned_to` при INSERT задачи заказчиком (ревью B-3).** `tasks_insert` после
правки 4.1 пустит заказчика (он `author=uid`, `is_approved`), но политика НЕ ограничивает
значение `assigned_to` — заказчик может назначить задачу произвольному uid. Изоляция не
пробита (задача в его проекте), но поведение нежелательное. **Серверная валидация:** `assigned_to`
обязан быть `null` ИЛИ присутствовать в `projects.executors` целевого проекта (с `userId != null`).
Реализация — BEFORE-триггер на `project_tasks` ИЛИ перенос создания задачи заказчиком в узкий
SECURITY DEFINER RPC `client_create_task(...)`. Решение — на этапе плана.

### 4.2. Приёмка — узкий RPC
Новый `client_set_task_status(p_task_id uuid, p_status text)` (SECURITY DEFINER):
- гейт `is_project_client(task.project_id)`;
- разрешённые переходы: `На проверке→Готово`, `На проверке→В работе`, `Готово→В работе`;
- иначе `raise exception`.

Причина отдельного RPC, а не расширения UPDATE-политики: обычный UPDATE заказчика-неавтора
не пускает (USING = author/assignee/admin/editor), и заказчик НЕ должен править
title/description/assignee чужой задачи — только статус приёмки.

### 4.3. Автоподтяг исполнителя
При создании задачи (заказчиком или сотрудником) `assigned_to` предзаполняется первым
элементом `projects.executors` с `userId != null`; пользователь может сменить/очистить.
Если очищено (null) — задача в пуле, исполнитель берёт кнопкой «Взять в работу».

## 5. Создание проектов — таблица `project_requests`

Прямое создание в `projects` невозможно безопасно (`owner_id` NOT NULL + marketplace
пускает заказчиков через дырявый `is_approved`). Заявки — отдельная сущность; материализует проект
**сотрудник** (решение владельца, §2 ось 6) — owner_id=сотрудник, NOT NULL чисто, сохранена модерация.

### 5.0. Почему owner_id ≠ заказчик (RLS-факт, проверено 2026-06-29)
RLS `projects`: `owner_id = auth.uid()` в `projects_select` (USING) даёт **полный доступ ко всей
строке**, в `projects_update`/`projects_delete` — право менять/удалять проект. Портальная проекция
`get_my_client_projects` намеренно скрывает `notes`/доли/себестоимость — но владелец строки обойдёт
её прямым `from('projects').select('*')`. Поэтому owner материализованного проекта = **сотрудник**,
а не заказчик. Заказчик видит свой проект только через `is_project_client`-проекцию.
**Заказчик заносит ВСЕ поля в заявке; приём сотрудником = один клик (без дозаполнения).**

### 5.1. Схема
```
project_requests (
  id uuid pk default gen_random_uuid(),
  client_id uuid not null references clients(id),     -- запись-заказчик автора
  created_by uuid not null references profiles(id),   -- аккаунт заказчика
  name text not null,
  description text,
  desired_deadline date,
  mode text not null default 'quick' check (mode in ('quick','detailed')),
  assignment_mode text not null default 'marketplace'
    check (assignment_mode in ('marketplace','assignee')),  -- v2: путь подбора
  desired_executor_id uuid references profiles(id),   -- v2: при assignment_mode='assignee'
  status text not null default 'Новая' check (status in ('Новая','Принята','Отклонена')),
  accepted_project_id uuid references projects(id),    -- проставляется при принятии
  created_at timestamptz default now()
)
```
- `mode` («оба режима»): `quick` (минимум полей) / `detailed` (расширенная форма).
- `assignment_mode` (v2): `marketplace` (выставить на поиск исполнителя) / `assignee` (заказчик
  назначил конкретного исполнителя `desired_executor_id`). Процесс единый — сотрудник принимает.

### 5.2. RLS `project_requests`
- SELECT: создатель (`created_by = auth.uid()`) ИЛИ сотрудник (`is_employee()`) ИЛИ админ.
  ⚠ для сотрудника гейт `is_employee()`, НЕ голый `is_approved` (иначе заказчики увидят чужие
  заявки). `is_employee()` вводится §4.0.
- INSERT: `am_i_client()` AND `created_by = auth.uid()` AND `client_id` принадлежит заказчику.
  ⚠ (ревью m-2) «принадлежит заказчику» = явный SQL в WITH CHECK, т.к. заказчик НЕ может SELECT
  из `clients` (политика `clients_select` = owner/admin): `EXISTS(SELECT 1 FROM public.clients
  WHERE id = client_id AND user_id = auth.uid())`.
- UPDATE/DELETE: только через RPC / админ.

### 5.3. RPC материализации
- `create_project_request(p_name, p_description, p_deadline, p_mode, p_assignment_mode,
  p_desired_executor_id default null, p_client_id default null)`:
  - гейт `am_i_client()`; `client_id` = переданный или единственная запись `clients` заказчика
    (§8.1 — сейчас всегда единственная, селектор в UI не нужен);
  - если `p_assignment_mode='assignee'` — валидировать, что `p_desired_executor_id` есть в списке
    `list_available_executors()` (защита от назначения произвольного uid);
  - шлёт in-app уведомление сотрудникам (паттерн заявки v2 из 0003).
- `accept_project_request(p_request_id)` (гейт: `is_employee()`/admin): создаёт `projects`
  (`owner_id`=принявший, `client_id`=из заявки, `name`/поля из заявки), ставит `status='Принята'`,
  `accepted_project_id`, уведомляет заказчика. **Ветвление по `assignment_mode`:**
  - `marketplace` → `visibility='marketplace'`, `stage='Поиск исполнителя'` → существующий
    broadcast-механизм уведомит исполнителей; берут через `take_project`.
  - `assignee` → `desired_executor_id` кладётся в `executors` jsonb + `project_members(editor)`,
    `stage='В работе'`. **⚠ НЕ вызывать `take_project` (ревью M-1):** внутри `take_project`
    `auth.uid()` = принявший сотрудник, а не `desired_executor_id` — назначит не того/упадёт.
    Выполнить вставку напрямую с `desired_executor_id`:
    - `executors`: `{"name": <profiles.name>, "userId": <desired_executor_id>}` — **формат точно
      как в `take_project`** (ревью M-3); idempotency-guard `WHEN executors @> jsonb_build_array(...)
      THEN executors ELSE ...||...` (не дублировать, если исполнитель уже там);
    - `executor` (текстовый) — пересобрать из `executors` (как в `take_project`);
    - `INSERT INTO project_members(project_id,user_id,role) VALUES(new_id, desired_executor_id,
      'editor') ON CONFLICT DO UPDATE SET role='editor'` (роль `editor` — единственная в БД).
- (опц.) `reject_project_request(p_request_id, p_reason)`.

### 5.4. RPC списка исполнителей (v2)
`list_available_executors()` (SECURITY DEFINER, гейт `am_i_client()`):
возвращает `{id, name, position}` для аккаунтов с ролью `employee` в `user_roles` И `approved=true`,
ИСКЛЮЧАЯ самого вызывающего (`profiles.id <> auth.uid()` — гибрид не назначает себя):
```sql
select p.id, p.name, p.position from public.profiles p
join public.user_roles ur on ur.user_id = p.id and ur.role = 'employee'
where p.approved = true and p.id <> auth.uid();
```
**НЕ возвращает** email/telegram/role/финансы. Узкая проекция — заказчик видит только имя и
должность для осознанного выбора. Это новая (но минимальная) поверхность раскрытия персданных
сотрудников; согласована владельцем (§2 ось 6, «оба пути»).
⚠ Порядок миграций (ревью m-3): `list_available_executors()` создаётся ДО `create_project_request()`,
т.к. последняя вызывает её для валидации `desired_executor_id`.

## 6. Фронт / UX

- **Портал «Мои заказы»**: индикатор задач на карточке проекта (счётчик), быстрый переход
  проект↔задачи (хотелка п.4).
- **Задачи проекта-заказа**: список карточек; «+ Поставить задачу» (автоподтяг исполнителя);
  открытие задачи → комментарии + ТЗ (видит/пишет); на «На проверке» — кнопки «Принять» / «Вернуть».
- **«Создать проект»** в портале: модалка выбора режима (Быстрая заявка / Подробный) →
  выбор пути подбора: «🔍 Найти исполнителя (маркетплейс)» / «👤 Назначить исполнителя» (при
  выборе — селект из `list_available_executors`) → `create_project_request`.
- **Сотрудник**: список входящих заявок + «Принять» (`accept_project_request`); бейдж «от заказчика»;
  в карточке заявки видно выбранный путь (маркетплейс / конкретный исполнитель).

## 7. Marketplace-утечка — РЕШЕНИЕ: чинить в фазе (владелец, 2026-06-29)

`is_approved()` не различает роли. Ветки `marketplace + is_approved()` в `projects_select` и
`can_access_project_comments` **пускают любого заказчика (approved) ко всем marketplace-проектам
целиком** через прямой `supabase.from('projects')` (в обход портала) — существующая утечка.

**Решение владельца:** закрыть в рамках этой фазы. Замена `is_approved()` → `is_employee()` в
существующих marketplace-ветках `projects_select` / `can_access_project_comments`.
⚠ **Регресс-внимание (ревью B-2, уточнено 2026-06-29):** это меняет поведение marketplace для всех —
после правки marketplace-проекты видят только сотрудники (не заказчики). Заказчик СВОЙ проект видит
через `get_my_client_projects()` (SECURITY DEFINER, обходит RLS, безопасная проекция).

**НЕ добавлять ветку `is_project_client(id)` в `projects_select`** (как может показаться очевидным
фиксом регресса): это вернуло бы заказчику ПОЛНЫЙ прямой доступ к строке `projects` (`select('*')`
→ `notes`/доли/себестоимость) в обход проекции — т.е. пробой §1, хуже регресса. Правильно: заказчик
получает проекты ТОЛЬКО через `get_my_client_projects()`.

**Обязательная проверка фронта (в план):** `fetchProjects()` (`App.jsx:368`, прямой
`from('projects').select('*')`) вызывается в bootstrap (`App.jsx:8578,8648`). Убедиться, что
клиентский портал (`isClientOnly`, `App.jsx:8778`) НЕ полагается на результат `fetchProjects` для
показа своих проектов, а берёт их из `get_my_client_projects`. Если заказчик где-то использует
прямой результат `fetchProjects` — переключить на RPC. Критерий — §10.

## 8a. Инварианты (зафиксированы по ревью безопасности, 2026-06-29)

- **И-1 (роли — уточнено данными 2026-06-29):** «заказчик» и «сотрудник» — НЕзависимые роли в
  `user_roles`. Аккаунт может быть **гибридом** `{client,employee}` (реальный кейс `6d95d516`) —
  тогда он и заказчик (свои клиентские проекты через `get_my_client_projects`), и сотрудник
  (`is_employee()=true` → видит заявки, принимает их, в списке исполнителей). «Чистый заказчик» =
  роль `client` БЕЗ `employee` → `is_employee()=false`, полная изоляция. План НЕ должен трактовать
  привязку к `clients` как «значит не сотрудник» — смотреть роль `employee`.
- **И-2 (ревью M-4 — закрыт существующим кодом):** аккаунт-заказчик `approved=true` гарантируется
  `set_client_user` (миграция 0005: при привязке ставит `approved=true`). Отдельной правки не
  требуется. **Верификация:** чек, что у всех `user_roles.role='client'` стоит `profiles.approved=true`.

## 9. Вне scope (YAGNI)

- Приватная переписка в задачах (→ будущий мессенджер).
- Флаг «видна заказчику» на задаче (заказчик видит все).
- Несколько исполнителей на одной задаче (`assigned_to` остаётся один).
- Множественный выбор записи `clients` в UI (де-факто одна; §8.1).

## 10. Критерии верификации

- Заказчик видит все задачи своих проектов и НЕ видит задачи чужих (после правки 4.1).
- Заказчик создаёт задачу, пишет комментарий, меняет ТЗ в своём проекте.
- Заказчик принимает (На проверке→Готово) и возвращает (→В работе); НЕ может двигать
  Новая→В работе и НЕ может править чужой `title`/`assignee`.
- Заказчик создаёт заявку (оба режима × оба пути подбора).
  - путь `marketplace` → после приёма сотрудником проект в «Поиск исполнителя», исполнителям
    приходит уведомление, исполнитель берёт через «Взять в работу».
  - путь `assignee` → после приёма выбранный исполнитель в команде и в `executors`, проект «В работе».
- `list_available_executors` возвращает только `{id,name,position}`; НЕ светит email/telegram/role.
- Заказчик НЕ может назначить `desired_executor_id` вне списка (валидация в `create_project_request`).
- **ТЗ (ревью B-1):** заказчик предлагает ТЗ к задаче, поставленной СОТРУДНИКОМ (после правки
  `propose_tz_version` + `is_project_client`), а не только к своей.
- **assigned_to (ревью B-3):** заказчик НЕ может назначить задачу uid вне `executors` проекта.
- **§7-регресс (ревью B-2):** после замены `is_approved`→`is_employee` в marketplace-ветках заказчик
  НЕ видит чужие marketplace-проекты прямым запросом, НО по-прежнему видит СВОЙ выставленный
  проект через `get_my_client_projects`; ветка `is_project_client` в `projects_select` НЕ добавлена;
  фронт-портал заказчика берёт проекты из RPC, не из `fetchProjects`; сотруднические
  marketplace-сценарии (`take_project`, список) не сломаны.
- **Инвариант И-2:** у всех `clients.user_id` стоит `profiles.approved=true`.
- Заказчик НЕ видит чужие `project_requests` и чужие проекты.
- Регресс: сотрудничьи сценарии задач/проектов не сломаны (баг 6.2 не вернулся).
