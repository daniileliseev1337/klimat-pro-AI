# Редизайн вкладки «Задачи» (TasksView) + workflow статусов + фото-отчёты

Дата: 2026-06-11 · Статус: дизайн согласован владельцем (brainstorm через Telegram, мокапы A/B → выбран B)

## Проблема

`TasksView`/`TasksBoard` (App.jsx ~4065–4229) — единственная вкладка, свёрстанная на голых
Tailwind-утилитах (`bg-zinc-800`, `bg-amber-500`), а не на токенах темы сайта (Card/Chip/BTN,
золото `#d4af37`, тёмный `#141414`). Список — голая `<table>` c minWidth 560. Итог: «на ПК как
набросок, на телефоне скомканно». Плюс владельцу нужны: автор на карточке, пошаговый workflow
статусов кнопками и фото-отчёты о выполнении.

## Решения владельца (зафиксированы)

1. **ПК-приоритет**; мобильная адаптация — отдельным проходом после.
2. **Доска (канбан) — главный вид**; список вторичен, но переверстать.
3. Стиль карточек — **вариант B** («просторные карточки», мокап `2026-06-11`, выбран из A/B).
4. Карточка: приоритет, проект, исполнитель (аватар+имя), **автор**, срок-светофор,
   **дата постановки**, индикаторы 💬 вопрос / фото.
5. Workflow-кнопки: взять в работу → отправить на проверку → «проверено, есть замечания
   (см. ТЗ)» → завершено. Автор=исполнитель → можно сразу «Готово» (без «На проверке»).
6. Фото-отчёты: **только фото**, прикладывать **в любой момент**, лимит **10 МБ**/файл.
7. Реализация **в 2 захода**: №1 доска+список+модалка+workflow (чистый фронт),
   №2 фото-отчёты (БД+edge+фронт).
8. «Отменена» на доске НЕ показывается — доступна через фильтр статуса.

## Заход №1 — доска, список, модалка, workflow

### Доска (стиль B)

- 4 колонки: Новая / В работе / На проверке / Готово. Заголовок колонки: цветная подложка
  (`<цвет>14`), имя цветом, счётчик. Цвета: Новая `#93c5fd`, В работе `#d4af37`,
  На проверке `#2dd4bf`, Готово `#6ee7a8` (согласовано со STAGE_META сайта).
- Drag-and-drop между колонками остаётся (нативный, как сейчас), но UI не предлагает
  недоступные переходы (см. workflow): чужую задачу нельзя бросить в «Готово», если ты не
  автор — сервер (`only_author_can_complete`) и так откажет, теперь и клиент не предлагает
  (toast с объяснением).
- Кнопка «+ задача» внизу каждой колонки — открывает модалку создания с предзаполненным
  статусом колонки.

### Карточка доски (сверху вниз)

1. Бейдж приоритета (таблетка): Высокий `#f8a3a3`, Обычный `#e8c860`, Низкий серый.
2. Название (15px, 700).
3. Строка проекта: 📁 имя проекта / 👤 «Личная задача».
4. Плашка «💬 Есть вопрос» (жёлтая, `#e8c860`) — если по задаче поднят вопрос (6.4b,
   нерешённый question-комментарий). ⚠ Текущий `get_tasks` этот флаг, скорее всего,
   не отдаёт — на планировании проверить и при необходимости добавить в RPC поле
   `has_open_question` (boolean, EXISTS-подзапрос к task_comments) — мелкая миграция.
5. Миниатюры фото-отчёта: до 3 + «+N» (заход №2; в заходе №1 секция не рендерится).
6. Разделитель; футер: аватар-кружок (инициалы, детерминированный цвет из PALETTE по
   userId) + имя исполнителя, под ним мелко «автор: <имя> · поставлена <дата>»;
   справа — срок-светофор.

**Срок-светофор:** 🟢 >3 дней до срока · 🟡 ≤3 дней · 🔴 просрочено («<дата> · −N дн»).
Без срока — серый прочерк. Порог жёлтого = 3 дня (константа, легко поменять).

### Workflow статусов (та же логика и в модалке, и в drag-and-drop)

Одна большая контекстная кнопка в модалке, по статусу и роли:

| Статус | Роль | Кнопки |
|---|---|---|
| Новая | исполнитель | **▶ Взять в работу** → В работе |
| В работе | исполнитель | **📤 Отправить на проверку** → На проверке; если автор=исполнитель — сразу **✓ Завершить** → Готово |
| На проверке | автор | **✓ Принять — завершено** → Готово · **↩ Есть замечания** → В работе |
| Готово / Отменена | — | кнопок нет |

- **«Есть замечания»**: обязательное текстовое поле замечания → текст уходит комментарием
  в обсуждение задачи (механика task_comments 6.4b), статус → «В работе», исполнителю
  push + inbox «Проверено, есть замечания — смотри ТЗ/обсуждение» (тип `task_status`
  с customText; новый тип в edge не нужен).
- Все переходы — через существующий RPC `set_task_status` (серверные правила не меняются).
- Каждый переход шлёт push + строку в центр уведомлений (существующий механизм notifyTask).
- Селект статуса в модалке **убирается** — заменён кнопками. «Отменить задачу» — отдельная
  второстепенная кнопка (автор; ghost-стиль) → статус «Отменена».
- Админ видит кнопки обеих ролей (как сейчас может всё).

### Список (вторичный вид)

- Вместо `<table>` — строки-карточки в теме сайта: бейдж статуса, приоритет, название,
  проект, исполнитель, светофор срока, дата постановки — в одну плотную строку.
- Клик — та же модалка. Сортировка: по сроку (дефолт, горящие сверху) / приоритету /
  дате постановки. Фильтры: проект / статус / приоритет (новый) / «только мои».
  Статус-фильтр «Отменена» — единственный путь увидеть отменённые.

### Модалка задачи

- Структуру 6.4b (версии ТЗ, diff, апрув, обсуждение) НЕ трогаем — только оформление
  под тему сайта (Card/BASE_INPUT/BTN) и шапка: крупный заголовок, бейджи статуса
  и приоритета, мета-строка «проект · исполнитель · автор · поставлена <дата> ·
  срок <светофор>», большая workflow-кнопка, секция «Фото-отчёт» (заход №2).
- Известный техдолг «TaskModal разросся» НЕ решаем здесь (не раздувать объём);
  новые куски — отдельными компонентами (TaskWorkflowButton, TaskCardBoard, TaskRowList).

### Шапка вкладки

Заголовок «Задачи» + подзаголовок-сводка («N активных · M требуют внимания», где
требуют внимания = просроченные + с вопросом); сегмент-переключатель Доска/Список
(золотая активная кнопка); справа «+ Новая задача» (BTN.primary).

## Заход №2 — фото-отчёты

### БД

```sql
create table task_photos (
  id uuid primary key default gen_random_uuid(),
  task_id uuid not null references project_tasks(id) on delete cascade,
  file_path text not null,          -- путь в Nextcloud: tasks/<task_id>/<filename>
  file_name text not null,
  file_size int not null,
  uploaded_by uuid not null references profiles(id),
  created_at timestamptz not null default now()
);
```

- RLS: SELECT — стороны задачи (через существующий хелпер `can_access_task`);
  INSERT — стороны задачи (uploaded_by = auth.uid()); DELETE — только uploaded_by.
- В realtime-publication НЕ добавляем (обновление по refetch при открытии модалки,
  как у версий/комментариев).
- SECURITY DEFINER нигде не нужен; новой миграции — `SET search_path` по правилу I-2.

### Хранилище и edge

- Файлы — в Nextcloud через **существующую** edge-функцию `nextcloud` (стриминг-механика
  как у файлов проектов): новые actions `task-photo-upload` / `task-photo-download` /
  `task-photo-delete`, путь `tasks/<task_id>/`. Права edge проверяет той же логикой
  под JWT юзера (метаданные в `task_photos` под RLS).
- Валидация на фронте И в edge: mime `image/jpeg|png|heic|webp`, ≤ 10 МБ.
- Удаление задачи: каскад строк есть (FK); файлы в NC подчищает edge при delete задачи —
  best-effort (как у файлов проектов; осиротевшая папка — косметика, прецедент уже есть).

### Фронт

- Модалка: секция «Фото-отчёт» — сетка миниатюр, «📷 Приложить фото» (любой момент,
  любая сторона задачи), клик — просмотр в полный размер (Blob/objectURL, как download
  файлов проектов), удалить может загрузивший.
- Карточка доски: до 3 миниатюр + «+N» (грузим метаданные batch'ем для видимых задач;
  миниатюра = тот же download, лениво).

## Не входит (явно)

- Мобильная адаптация вкладки — отдельный проход после ПК-версии.
- Рефакторинг TaskModal на подкомпоненты сверх необходимого.
- Новые типы push в edge (переиспользуем task_status + customText).
- Изменение серверных правил статусов (set_task_status как есть).
- Видео/документы во вложениях — только фото.

## Критерии приёмки

Заход №1: доска и список визуально в теме сайта (без zinc-классов в TasksView/TasksBoard);
на карточке видны приоритет/проект/исполнитель/автор/дата постановки/светофор срока;
workflow-кнопки работают по таблице ролей; «есть замечания» требует текст и кладёт его
в обсуждение + push; селекта статуса в модалке нет; «Отменена» видна только фильтром;
vitest зелёный, build зелёный.

Заход №2: фото грузится (≤10 МБ, только image/*), видно миниатюрами в модалке и на
карточке, скачивается в полный размер, удаляется загрузившим, RLS-скрипт подтверждает
изоляцию (чужой не видит/не грузит), E2E upload→view→delete пройден.
