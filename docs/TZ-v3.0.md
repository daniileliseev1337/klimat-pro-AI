# Техническое задание для Claude Code: daniil-dashboard v3.0

## Назначение этого документа

Это полное техническое задание на развитие существующего проекта daniil-dashboard до версии 3.0. Документ предназначен для передачи агенту Claude Code, который будет вести разработку на локальном ПК. Документ описывает текущее состояние системы, цели переработки, целевую архитектуру, требования к каждому модулю и критерии готовности.

Документ следует читать целиком до начала любых работ. Реализацию вести поэтапно в порядке разделов 6.1–6.7. После каждого этапа сверяться с критериями готовности соответствующего раздела.

## 1. Контекст и текущее состояние

### 1.1. Что такое daniil-dashboard

Личный рабочий центр для управления инженерными проектами (ОВиК, BIM, Revit), финансами и аналитикой. Используется небольшой внутренней командой (владелец и несколько исполнителей). Заказчики в системе не авторизуются, они хранятся как данные.

### 1.2. Текущий технический стек (версия 2.0)

Фронтенд: React 18 плюс Vite, один основной файл src/App.jsx размером около 6200 строк.
Бэкенд: Supabase, размещён в облаке в регионе eu-west-1 (Ирландия). Включает PostgreSQL, авторизацию, Row Level Security, Edge Functions.
Хостинг фронта: Vercel, домен daniil-dashboard.vercel.app.
Файловое хранилище: Yandex Disk через Edge Function и REST API.
Дизайн: тёмная тема Black and Gold.

### 1.3. Что уже реализовано в версии 2.0

Управление проектами с четырьмя режимами видимости (приватный, командный, избранные пользователи, маркетплейс). Маркетплейс свободных проектов с возможностью взять проект в работу. Комментарии к проектам с разделением на открытые и решённые. Telegram-бот для уведомлений (взятие проекта, приглашение в команду, комментарии, назначение исполнителя). Файловое хранилище проектов на Yandex Disk. Базовая авторизация с одобрением новых пользователей администратором.

### 1.4. Реальные проблемы версии 2.0, которые решает версия 3.0

Проблема первая, главная. Сайт работает нестабильно, нормально функционирует только при включённом VPN. Причина установлена точно: IP-адреса облачного Supabase (зарубежный AWS, Ирландия) и Vercel фильтруются российскими провайдерами. Без VPN соединение рвётся или замедляется.

Проблема вторая. Нет единой панели задач. Сейчас задачи не существуют как сущность. Нужен слой, который видит задачи по всем проектам, позволяет их быстро принимать, создавать, проваливаться внутрь, изменять и подтверждать.

Проблема третья. Оплаченные проекты постоянно видны и засоряют список. Их нужно автоматически скрывать или архивировать.

Проблема четвёртая. Дашборд мёртвый. Не агрегирует доходы и расходы, не сводит данные в плашки, создан формально, смотреть на нём нечего.

Проблема пятая. Файловое хранилище на Yandex Disk неудобно и неинтуитивно. Работа с ним сводится к загрузке и скачиванию, нельзя редактировать файлы на месте.

## 2. Цели версии 3.0

Цель 1. Устранить зависимость от VPN. Перенести всю систему на локальный ПК с публичным российским IP, чтобы доступ шёл по обычному HTTPS без обхода блокировок.

Цель 2. Создать единый слой задач как самостоятельную сущность над всеми проектами.

Цель 3. Сделать дашборд живым, реально сводящим финансы и состояние проектов.

Цель 4. Заменить Yandex Disk на локальное файловое хранилище с возможностью редактирования на месте.

Цель 5. Добавить MCP-слой, через который Claude может читать структуру системы и выполнять действия (создавать и менять ТЗ, подтверждать, комментировать) с подтверждением и проверкой прав.

Цель 6. Сохранить принцип интуитивности. Любой человек, зайдя впервые, понимает что и как работает без инструкций и лекций.

## 3. Целевая архитектура версии 3.0

### 3.1. Где что размещается

Всё размещается на одном домашнем ПК, который работает постоянно. Это единственный сервер. На нём крутятся:

Self-hosted Supabase через Docker. Заменяет облачный Supabase. База данных, авторизация, RLS, Edge Functions работают локально.
Веб-сервер фронта. Caddy или nginx отдаёт собранный React, выступает обратным прокси и завершает TLS.
Nextcloud. Локальное файловое хранилище, заменяет Yandex Disk.
MCP-сервер. Отдельный сервис, оборачивает действия системы в инструменты для Claude.

Облачный Supabase и Vercel убираются полностью.

### 3.2. Сетевой доступ

Сценарий зависит от типа IP домашнего провайдера. Перед стартом этот пункт необходимо уточнить у владельца.

Если IP белый (публичный). Caddy на ПК принимает HTTPS напрямую. Используется DDNS для привязки домена к меняющемуся адресу и Let's Encrypt для сертификата. Внешняя точка входа не требуется. Это бесплатный и предпочтительный вариант.

Если CGNAT (серый IP). Нужна минимальная внешняя точка входа-форвардер с публичным адресом. ПК сам открывает исходящее TLS-соединение к форвардеру через обратный туннель (FRP или reverse SSH tunnel), форвардер пробрасывает входящие запросы обратно на ПК. Форвардер не хостит ничего, только перенаправляет. Вся мощность и данные остаются на ПК.

Важно про обход блокировок. На ПК часто включены VPN и сервис zapret. Они не должны мешать работе сервера. Принципы:

VPN настраивается в режиме split-tunnel. Через VPN идёт только исходящий трафик обхода (Claude API, зарубежные сервисы). Трафик раздачи сайта и туннель к форвардеру идут напрямую, минуя VPN.

zapret работает в режиме списка хостов. Он трогает только конкретные заблокированные сайты и не касается соединений с собственным сервером и российским форвардером.

Обратный туннель устойчив к zapret, потому что это исходящее TLS-соединение, которое выглядит как обычный HTTPS, а не как VPN. Этим он принципиально отличается от ранее использованного Tailscale на базе WireGuard, который ломался из-за конфликта с VPN и zapret.

### 3.3. Файловое хранилище

Nextcloud на ПК. Активная рабочая папка доступна через веб-интерфейс и через клиент синхронизации, который показывает её как локальную папку с редактированием на месте. Холодный архив старых проектов (большая часть из примерно 40 ГБ) хранится отдельной папкой и подключается по необходимости, не висит у всех постоянно.

Замечание про тяжёлые файлы. Для крупных моделей Revit и CAD работа напрямую по сети может тормозить, а Revit плохо работает с сетевыми путями для центральных моделей. Для документов, PDF, Excel, изображений всё работает хорошо. Тяжёлые модели предпочтительно держать через локальную синхронизацию, а не прямой сетевой доступ.

### 3.4. Принцип разделения ролей машин

Домашний ПК отвечает за сетевую раздачу сервиса внутри российского интернета и за хранение данных. VPN и zapret на этой же машине обслуживают исходящий обход цензуры. Это две противоположные задачи, и они разведены: сервер доступен изнутри РФ по белому IP, обход работает наружу через split-tunnel. Мощность ПК для нагрузки небольшой команды избыточна, узким местом является только сетевая стабильность, которая обеспечивается российским IP.

## 4. Функциональные требования

### 4.1. Миграция бэкенда на self-hosted Supabase

Развернуть Supabase через Docker на ПК. Перенести существующую схему базы: все таблицы, функции, RLS-политики, Edge Functions из облачного проекта. Перенести данные (проекты, пользователи, профили, комментарии, файлы, настройки Telegram). Обновить переменные окружения фронта на локальный адрес Supabase. Проверить что авторизация, RLS и Edge Functions работают локально идентично облаку.

### 4.2. Слой задач (таск-панель)

Задача это самостоятельная сущность, не подзадача проекта. Задача может быть связана с проектом, но может существовать и без привязки. Слой задач это надстройка, которая видит задачи по всем проектам сразу.

Требования к панели задач:

Единый список всех задач по всем проектам с фильтрами (по проекту, по статусу, по исполнителю, по дедлайну).
Быстрое создание задачи в пару действий.
Быстрое принятие задачи в работу.
Проваливание внутрь задачи для просмотра и редактирования полного содержания.
Изменение и дополнение задачи и её ТЗ.
История изменений задачи с показом что именно поменялось в ТЗ (формат diff, видно добавленное и удалённое).
Подтверждение (апрув) изменений или статусов.
Мгновенные уведомления об изменениях статуса, обновлении ТЗ и о том что именно добавилось. Уведомления через уже существующий Telegram-бот.

Принцип интерфейса. Всё интуитивно, без обучения. Человек заходит и сразу понимает структуру и действия. Референсы для идей (не для копирования): KanVibe (доска для AI-агентов с автодвижением задач по статусам, мгновенные уведомления, diff-просмотр изменений в браузере) и TaskView (история изменений каждой задачи, гранулярные права, real-time синхронизация). Взять принципы и удачные решения, встроить в существующую систему, не плодя второй продукт и второй вход.

### 4.3. Переработка дашборда

Текущий дашборд не сводит данные. Переделать в живой сводный экран. Минимальный набор того что видно при открытии:

Текущий баланс.
Доходы против расходов по периодам (с агрегацией по месяцам или выбранному периоду).
Портфель активных проектов.
Ближайшие дедлайны.

Дашборд должен реально подтягивать и агрегировать данные о тратах и доходах, а не показывать пустые или статичные плашки. Конкретный набор метрик и графиков согласовать с владельцем перед реализацией этого раздела.

### 4.4. Скрытие оплаченных проектов

Проекты на стадии Оплачен автоматически скрываются из основного списка или уходят в архив. Должна быть возможность при желании показать архив. Это небольшая правка фильтрации списка проектов.

### 4.5. Замена Yandex Disk на Nextcloud

Убрать интеграцию с Yandex Disk (Edge Function yandex-disk и связанные вызовы). Подключить файловое хранилище через Nextcloud на ПК. Файлы проекта хранятся в Nextcloud, привязка к проекту сохраняется в базе. Поддержать просмотр списка файлов, загрузку, скачивание, удаление, и по возможности редактирование на месте через интерфейс Nextcloud или клиент синхронизации.

### 4.6. MCP-слой для Claude

Отдельный MCP-сервер, размещён на ПК рядом с Claude Code. Сервер знает структуру системы и оборачивает действия в инструменты, доступные Claude. Минимальный набор инструментов:

Чтение: список проектов, детали проекта, список задач, детали задачи, текущее ТЗ.
Запись: создать ТЗ, изменить ТЗ, создать задачу, изменить задачу, подтвердить, добавить комментарий.

Через эти инструменты любой человек с доступом может попросить Claude сформировать ТЗ, внести изменение, что-то подтвердить, и это запишется прямо в систему.

Требования безопасности MCP-слоя:

Все операции записи выполняются только с явным подтверждением, Claude не перезаписывает ТЗ молча.
Проверка прав: MCP-сервер действует с понятной идентичностью и не может трогать проекты вне разрешённого доступа.
Поскольку бэкенд уже на Supabase с готовыми RPC-функциями, инструменты MCP оборачивают эти RPC.

Референс реализации: TaskView имеет нативный MCP-сервер для Claude, посмотреть как устроено у них.

## 5. Нефункциональные требования

Стабильность. Система должна работать без VPN со стороны пользователя. Доступ по обычному HTTPS.
Интуитивность. Интерфейс понятен без инструкций. Это требование уровня дизайна, закладывается с самого начала, а не прикручивается потом.
Преемственность. Переиспользовать существующую авторизацию, пользователей, проекты, права, Telegram-уведомления. Не создавать вторую параллельную систему.
Безопасность. Backend-сервисы не торчат в интернет прямыми портами. Наружу открыты только 80 и 443 через обратный прокси. Запись через MCP с подтверждением и проверкой прав.
Бесплатность инфраструктуры по возможности. При белом IP вся система работает на ПК без платных сервисов. Платный форвардер только если подтвердится CGNAT.

## 6. Этапы реализации и критерии готовности

Реализацию вести строго по этапам. К следующему этапу переходить только после выполнения критериев готовности предыдущего.

### 6.1. Этап подготовки инфраструктуры

Уточнить тип IP домашнего провайдера (белый или CGNAT), проверив раздел WAN в роутере.
Развернуть Docker на ПК. Развернуть self-hosted Supabase. Настроить Caddy или nginx. При белом IP настроить DDNS и Let's Encrypt. При CGNAT настроить обратный туннель к форвардеру.
Критерий готовности: локальный Supabase отвечает, сайт-заглушка открывается по домену снаружи без VPN.

### 6.2. Этап миграции бэкенда

Перенести схему базы, данные, Edge Functions с облака на локальный Supabase. Переключить фронт на локальный адрес.
Критерий готовности: текущий функционал версии 2.0 полностью работает на локальном бэкенде, авторизация и права сохранены, данные на месте.

### 6.3. Этап файлового хранилища

Развернуть Nextcloud. Перенести файлы с Yandex Disk. Переключить функционал файлов проекта на Nextcloud. Убрать интеграцию Yandex Disk.
Критерий готовности: файлы проектов доступны, загружаются, скачиваются, удаляются, редактируются на месте, Yandex Disk больше не используется.

### 6.4. Этап слоя задач

Спроектировать сущность задачи и её связь с проектами. Реализовать единую панель задач со всеми требованиями раздела 4.2. Подключить уведомления через Telegram.
Критерий готовности: задачи создаются, принимаются, редактируются, видна история изменений ТЗ в формате diff, приходят мгновенные уведомления, интерфейс понятен без обучения.

### 6.5. Этап переработки дашборда

Согласовать с владельцем набор метрик. Реализовать живой сводный дашборд с агрегацией доходов и расходов, активных проектов и дедлайнов.
Критерий готовности: дашборд при открытии показывает реальные сведённые данные, а не пустые плашки.

### 6.6. Этап скрытия оплаченных проектов

Реализовать автоскрытие или архивацию проектов на стадии Оплачен с возможностью показать архив.
Критерий готовности: оплаченные проекты не засоряют основной список, архив доступен по запросу.

### 6.7. Этап MCP-слоя

Реализовать MCP-сервер с инструментами чтения и записи из раздела 4.6, с подтверждением и проверкой прав.
Критерий готовности: Claude через MCP читает структуру и по запросу человека формирует и меняет ТЗ, подтверждает, комментирует, всё с подтверждением и в рамках прав.

## 7. Принципы работы для агента Claude Code

Думать до того как писать код. При неуверенности задавать вопрос владельцу, не выдумывать.
Простота. Минимум кода и правок, без абстракций ради одноразовых задач.
Хирургические правки. Каждое изменение под конкретную задачу, соседний рабочий код не трогать без необходимости.
Цели и проверка. Критерии готовности определены для каждого этапа выше, сверяться с ними.
Честность. Возражать при наличии оснований, не поддакивать.

## 8. Открытые вопросы перед стартом

Тип IP домашнего провайдера: белый или CGNAT. От этого зависит нужен ли форвардер. Проверяется в роутере, раздел WAN.
Точный набор метрик и графиков дашборда (раздел 4.4 и 6.5). Согласовать с владельцем.
Состав полей сущности задачи. Согласовать при проектировании этапа 6.4.
